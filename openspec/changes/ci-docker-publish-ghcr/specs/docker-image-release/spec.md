## ADDED Requirements

### 需求:CI 必须将容器镜像发布到 GHCR

系统必须在约定触发条件下使用根目录 `Dockerfile` 构建 OCI 镜像，并推送到 GitHub Container Registry，且推送操作必须使用具备 `packages: write` 权限的自动化凭证完成。

#### 场景:通过版本标签发布可复现镜像

- **当** 维护者向本仓库推送与语义化版本匹配的 Git 标签（例如以 `v` 为前缀的标签）
- **那么** CI 必须成功构建镜像并以该标签作为镜像标签之一推送到 GHCR

#### 场景:镜像坐标可预测

- **当** 使用者需要拉取本项目的官方镜像
- **那么** 文档或工作流产出中必须给出形如 `ghcr.io/<owner>/<repo>` 的镜像坐标说明（`owner`/`repo` 与 GitHub 仓库一致且符合 GHCR 命名惯例）

### 需求:Docker Compose 必须与已发布镜像路径一致

`docker-compose.yml` 必须同步体现“使用已发布镜像部署”的方式：通过 `image` 字段引用 GHCR 上的镜像，并保留或并列说明本地 `build` 开发路径，使同一文件或文档不误导用户以为只能本地构建。

#### 场景:生产类部署可仅拉取镜像

- **当** 部署方希望不克隆构建上下文而直接运行容器
- **那么** 其必须能够依据仓库中的 Compose 片段或注释，使用 `docker compose pull` / `image:` 引用完成启动

#### 场景:本地开发仍可构建

- **当** 开发者需要在本地修改代码并立即用 Compose 验证
- **那么** 仓库必须仍提供基于 `build: .` 或等效方式的本地构建路径，且不与“仅拉镜像”路径互相破坏（例如通过注释说明、分离 service 或 Compose profile）
