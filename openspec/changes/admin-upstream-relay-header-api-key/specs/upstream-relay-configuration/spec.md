## 新增需求

### 需求:管理配置的上游基址

系统必须从持久化全局设置解析上游中转站的**基础 URL**，并在所有模型路由中用于拼接 OpenAI/Anthropic/Gemini 等路径；该基址必须在管理面板中可查看与修改。**禁止**使用环境变量（例如 `PROXY_TARGET_URL`）作为全局基址来源或回退。

#### 场景:管理员保存中转站地址

- **当** 管理员在管理面板中设置「中转站地址」并成功保存
- **那么** 后续上游请求必须使用保存后的基址（除非单条模型映射显式覆盖 `target_url`）

#### 场景:未配置持久化基址时不得静默使用环境变量

- **当** 持久化全局「中转站地址」为空，且当前模型映射未提供非空 `target_url`
- **那么** 系统必须拒绝执行上游转发并返回明确错误，**禁止**使用环境变量补全基址

### 需求:从客户端请求解析上游密钥

系统必须使用当前 Cursor（或 HTTP 客户端）发往本服务的请求头中的鉴权信息作为**上游 API 密钥材料**，再按目标后端类型生成实际上游请求头。当请求中**能够**解析出密钥材料时，必须使用该材料。当请求中**不能**解析出密钥材料时，唯一回退为模型映射中该 Cursor 模型的 `api_key`。**禁止**使用管理面板全局 `proxy_api_key` 或环境变量 `PROXY_API_KEY` 作为密钥来源或回退。

#### 场景:由 Authorization 承担上游密钥

- **当** 部署肯定不配置 `ACCESS_API_KEY`，且请求包含 `Authorization: Bearer <token>`
- **那么** 系统必须使用 `<token>` 作为上游密钥材料向中转站发起请求

#### 场景:请求未携带可用上游密钥时的回退

- **当** 按上述规则无法从请求头得到上游密钥材料
- **那么** 系统必须使用模型映射中的 `api_key`（若有）；若映射也无密钥则上游请求无凭证。**禁止**使用管理面板全局 `proxy_api_key` 或环境变量 `PROXY_API_KEY`
